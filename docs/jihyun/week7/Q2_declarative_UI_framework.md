# Jetpack Compose가 선언적 UI 프레임워크라고 불리는 이유

Jetpack Compose는 개발자가 상태를 기반으로 UI가 무엇을 표시해야 하는지를 표현할 수 있도록 하는 선언적 UI 프레임워크

## Compose의 주요 특징

- `@Composable` 어노테이션이 달린 함수를 사용하여 UI 컴포넌트를 정의하는 것으로, 선언적 UI의 첫 번째 원칙
- 상태 주도 UI (State-Driven UI): 개발자는 특정 상태에 대해 UI가 어떻게 보여야 하는지만 정의하고, 프레임워크가 상태 변경 시 UI를 자동으로 업데이트
    - `remember`와 같은 Compose Runtime의 기능을 통해 컴포저블의 상태와 생명주기를 효율적으로 관리
    - 상태 변경 시, `recomposition`을 트리거하여 영향을 받는 컴포넌트만 업데이트하고 최신 데이터를 반영
- 개발자는 모델 데이터를 UI 컴포넌트에 직접 바인딩할 수 있어, XML처럼 추가적인 데이터 바인딩 메커니즘이나 복잡한 아키텍처 패턴이 필요 없이 UI 개발을 크게 단순화
- `@Composable` 함수는 본질적으로 멱등성(Idempotence)을 가지므로, 동일한 입력 매개변수에 대해 호출 횟수에 관계없이 항상 동일한 UI 결과를 생성하여 예측 가능하고 안정적인 UI 렌더링을 가능하게 함

## 전통적인 명령형 XML UI와의 차이점

| **특징** | **Jetpack Compose (선언적)** | **XML 기반 (명령형)** |
| --- | --- | --- |
| **UI 정의** | Kotlin 언어 자체 내에서 `@Composable` 함수로 UI와 상태 정의를 밀접하게 통합. | XML 레이아웃 구조 및 속성 정의와 Kotlin/Java를 사용한 UI 업데이트 명령형 코드가 별도로 구현됨. |
| **UI 업데이트** | 상태가 변경되면 UI가 **자동으로** 업데이트되도록 프레임워크가 보장 (State-Driven UI). | 개발자가 `findViewById()` 등을 통해 뷰를 찾고 **수동으로** 속성을 변경하는 명령형 핸들러가 필요함. |
| **데이터 처리** | 함수 매개변수를 통해 데이터를 UI에 **직접 바인딩**하여 코드가 더 응집력 있음. | UI와 애플리케이션 로직 사이에 수동적인 동기화 또는 복잡한 워크플로우가 요구됨. |
